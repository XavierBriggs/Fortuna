# Area 5 — Vendor/Venue Coupling

## Summary

The adapter boundary is **largely clean and demo-paper-ready**. The `Venue` trait (`fortuna-venues/src/lib.rs:91`) accepts only `GatedOrder`; strategies and risk gates consume neutral domain types (`Cents`, `MarketId`, `GatedOrder`, `Fill`) and have zero direct imports from any vendor SDK or DTO module. The biggest risk is a cluster of deliberate, documented seam violations where `KalshiMarket` and `KalshiMarketStatus` DTOs cross the adapter boundary into `fortuna-live`'s F7 weather plug-in code and its smoke tests — this is a conscious design choice (the bucket-matching seam must see raw bracket fields not present in the domain `Market` struct), but it binds the weather discovery pipeline specifically to Kalshi. A secondary concern is that the sim fee schedule is loaded under the hardcoded key `"kalshi"` in config, coupling the sim world's fee pricing to the Kalshi fee section even when no Kalshi transport is present. Both issues are bounded and clearly named; they block a clean second-venue addition but do not block the paper demo.

## Trace / narrative

### Where the boundary holds (the good)

The `Venue` trait (`fortuna-venues/src/lib.rs:91`) is the correct seam:

- `place` accepts only `GatedOrder` (a type-level I1 enforcement).
- `markets()` returns `Vec<Market>` (a domain type in `fortuna-venues/src/types.rs`), not `Vec<KalshiMarket>`.
- `fills_since` returns `FillPage` containing domain `Fill` objects.
- `settlements_since` returns `SettlementPage` / `SettlementNotice` — domain types, though `SettlementNotice.detail` is a `serde_json::Value` carrying the raw venue payload for audit purposes (acceptable: labelled "data, never instructions" at `types.rs:72`).

Strategies (`mech_structural.rs`, `perp_event_basis.rs`, `perp_event_basis_v2.rs`) operate exclusively on domain types: `MarketId`, `OrderBook`, `PriceLevel`, `Cents`, `Contracts`, `Side`, `Action`. Zero Kalshi imports appear in any strategy module (`grep` of `use fortuna_venues::kalshi` in `fortuna-runner/src/` returns only test code and a test-only inline module at `runner.rs:3581`).

Risk gates (`fortuna-gates`) are completely clean — the only "kalshi" string is the literal venue id used in a test at `pipeline.rs:216` (`HaltScope::Venue("kalshi".into())`), which is correct usage: halt scopes take the venue id string as a runtime value, not a compile-time import.

The `fortuna-exec` crate has zero Kalshi references. The `fortuna-cognition` source modules have no code-level Kalshi imports (only doc-comments referencing Kalshi as domain context). `fortuna-ledger` uses `"kalshi"` only in test data fixtures as a venue id string value.

### Where the boundary leaks (the bad)

The F7 weather discovery seam (`fortuna-live/src/aeolus_venue.rs`) is the most significant coupling point. The function `market_to_bucket` at `aeolus_venue.rs:111` takes `&KalshiMarket` (a Kalshi DTO) and reads `m.strike_type`, `m.floor_strike_int()`, `m.cap_strike_int()` — fields that are Kalshi-specific and absent from the domain `Market` struct. This function is called from `daemon.rs:2362`, which also reads `m.status == fortuna_venues::kalshi::dto::KalshiMarketStatus::Active` directly at `daemon.rs:2360`, bypassing the domain `MarketStatus` enum.

The `WeatherMarketSource` trait at `fortuna-venues/src/kalshi/weather.rs:34` returns `Result<Vec<KalshiMarket>, VenueError>`. Because this trait is how the daemon injects a testable weather discovery source, the vendor DTO is the currency of the whole F7 path — from trait definition through `DiscoveryWiring.weather_source` field type to the production daemon and every test stub.

In tests (`aeolus_bucket_match.rs:19–50`, `daemon_smoke.rs:3198–3242`), the mock objects are `Vec<KalshiMarket>` deserialized from recorded JSON fixtures. This is technically appropriate for fixture-grounded tests but means the test interface is vendor-specific.

The `aeolus_venue.rs:165` hardcodes `venue: "kalshi".to_string()` in the edge proposal, making every auto-confirmed weather edge permanently labelled "kalshi" regardless of which live venue the day-set came from.

The sim composition at `daemon.rs:306` requires `[fees.kalshi]` in config. This is an implicit coupling: the sim world and the Kalshi demo use the same fee key, so a sim-only operator must still provide Kalshi fee parameters. The error message `"[fees.kalshi] missing (the sim world prices with it)"` acknowledges the coupling explicitly.

The killswitch (`fortuna-killswitch/src/main.rs`) correctly imports and uses `KalshiVenue` and `KineticsAdapter` because its job is to cancel open orders — it must know the concrete venue type to construct the transport. This is appropriate for a per-venue operational tool.

`PaperLiveVenue` (`fortuna-paper/src/paper_live.rs`) stores `KalshiReadClient` as a named field. This is architecturally appropriate for the current single-venue paper demo but would need abstraction for a second live venue. The `PaperLiveVenue` itself is a Kalshi-specific composite by documented design; the `Venue` trait it implements (`impl Venue for PaperLiveVenue`) is generic.

### Second-venue addition assessment

Could a second venue be added **without** editing strategy or gate logic? **Yes** — strategies and gates are clean. Could it be added without editing `fortuna-live`? **No** — the F7 weather discovery path, `DiscoveryWiring`, and the sim fee loading all have Kalshi-specific coupling points in `daemon.rs` and `aeolus_venue.rs`. A second binary-market venue (e.g. Polymarket) requires editing those paths.

### Fixture-grounding discipline

The codebase follows the "no invented venue behavior" rule. The Polymarket adapter (`fortuna-venues/src/polymarket/mod.rs`) is a `FixtureGated` stub that refuses all operations with `VenueError::FixtureGated` and a refusing fee model. The Kinetics adapter is implemented against recorded fixtures. This is good coupling discipline: the boundary code is fixture-driven, not spec-invented.

## Self-adversarial pass

**Is the KalshiMarket leakage actually a problem for the paper demo?** No — the demo is Kalshi-only. The leakage is a real multi-venue impediment but `BLOCKS` nothing for demo day. Severity P2 (delivery risk for a second venue) is correct; P1 would be wrong.

**Is the `fees.get("kalshi")` coupling really a code problem?** It could be read as config-only — the key is a string literal in TOML and in two `get()` calls. If the sim fee model should match Kalshi fees (intentional design), the coupling is correct and the only oddity is the key name. However, it does mean a sim-only operator who reads the error message is confused ("why do I need a kalshi section?"). The framing as P2/PARK is appropriate.

**Did I miss any strategy/gate coupling?** I ran `grep -rn "kalshi\|Kalshi"` across `fortuna-runner/src/` (strategies), `fortuna-gates/` (gates), `fortuna-exec/` (execution), and `fortuna-core/` (domain types). The only non-test, non-comment Kalshi references in strategy code are in `mech_structural.rs:10` (a comment explaining the YES/NO book convention), `perp_event_basis.rs:8–9` (doc comment, "KXBTC"), and `perp_event_basis_v2.rs:362` (comment about fee coefficient). No code-level vendor coupling in strategies.

**Could the `oracle_type: "kalshi_rulebook"` stub cause a real bug?** The stub market is overwritten by `venue.markets()` on the first tick. If the first tick fails (network outage), the runner would see the stub metadata. A strategy that keys on `oracle_type` would get wrong data for the bootstrap window. No current strategy reads `settlement.oracle_type`, so this is theoretical. Severity P3 is correct.

**Did I miss the sim comment in `sim.rs` ("pair-mint matching like Kalshi")?** Yes — I noted this in the sim.rs read (`sim.rs:8`). This is a comment explaining the yes/no mirror-book mechanism; it uses "Kalshi" as an example/model, not as a code dependency. The sim is fully independent of the Kalshi adapter in its implementation.

## Open questions for the Lead

1. **Domain extension vs. seam type**: The right long-term fix for the `KalshiMarket` leakage is extending the domain `Market` struct with optional bracket fields (or a `TemperatureBracketMarket` wrapper). Should this happen before the demo or is it a Phase 3 task? The current state is demo-safe but blocks weather discovery for any second venue.

2. **`fees.get("kalshi")` in sim**: Was the intent to always price the sim world with Kalshi fee parameters (in which case, the coupling is intentional and the key name should just be documented), or should the sim support a configurable fee key? This determines whether this is a rename or a docs-only fix.

3. **`WeatherMarketSource` trait location**: The trait currently lives under `fortuna_venues::kalshi`. If there is appetite to move it to a neutral namespace before the demo, it is a pure refactor (the trait interface does not change behavior). Is there a migration window for this?

4. **Kinetics base URL at `external-api.kalshi.com`**: The Kalshi-branded domain serving the Kinetics perp API (`funding_poller.rs:59`) is correct today per recorded research. Is there a known timeline for Kinetics to migrate to a separate domain that would invalidate this constant?
