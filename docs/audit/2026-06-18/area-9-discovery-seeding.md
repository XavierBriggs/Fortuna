# Area 9 — Discovery, Seeding & the Signal→Event Pipeline

## Summary

Discovery is structurally sound — budget caps, unscoreable guards, dedup, and the market-back/world-forward split are all present and tested. For the paper demo it is not blocking in the direct money path. However the operator's experience is genuinely a mess: all 20 world-forward watch events in `fortuna_demo` have zero beliefs attached, 16/20 are unscoreable because the model emits human-readable resolution-source strings that can never match the short registry source IDs (`rss_fed_press`, `nws_alerts_ny`, etc.), and there is no operator-facing CLI command to seed or inspect events. The category vocabulary is chaotic (12 distinct spellings for macro/weather across 20 watch events). Market-back discovery silently forwards an always-empty `existing_events` slice, disabling the match-before-create path. The dedup window is a raw `LIMIT 200` query with no guaranteed coverage of old events. None of these breaks the paper loop's money path directly, but the calibration substrate (beliefs) is not accruing on world-forward events as designed.

## Trace / Narrative

### Signal ingestion (confirmed working)

~28,547 signals exist in `fortuna_demo`. The last 24 hours show types `nws.afd` (8,192), `release_scheduled` (4,382), `rss.item` (280), `aeolus.forecast` (186), `nws.alert` (73), `nws.cli` (70). The ingestion funnel — `SourceRegistry` loaded from `source_registry` at boot (`ingestion.rs:253–278`), content-hash dedup via `DedupIndex`, `SignalsRepo::insert` with the append-only trigger — is functioning correctly. The `type` column is the DB name; the repo layer aliases it to `kind` in Rust (`repos.rs:911`). The `recent_by_kind` query (`repos.rs:904–920`) filters by `type = ANY($1)`, which correctly handles array binding.

### World-forward discovery (confirmed broken for non-Aeolus events)

The loop is wired and fires every segment when `[discovery] enabled = true` and `signal_kinds` is non-empty. The structured-output fix (`decide_structured` + `watchlist_schema`, committed in `16fba55`) resolved the "watchlist body violated the contract" error that had completely silenced the loop. Candidates now persist as `watch:` events.

However, the **unscoreable test at `discovery.rs:688`** does an exact-match lookup: `registry.get(&entry.resolution_source)`. The registry source IDs are short machine names (`nws_alerts_ny`, `rss_fed_press`, `calendar_bls`, `aeolus_knyc`, etc.). The model — prompted only with the generic system charter (`SYNTH_MIND_SYSTEM_CHARTER: "You synthesize calibrated probabilistic beliefs..."`) — consistently emits human-readable resolution descriptions ("Federal Reserve FOMC statement", "NWS observed data", etc.). These never match any registry key. `unwrap_or(true)` fires, the candidate is marked `unscoreable = true`, and **no belief is persisted for it** (`discovery.rs:739–750`). The unscoreable rule persists a `watch:` event row for tracking but refuses its belief.

Confirmed in DB: 20 watch events, 16 unscoreable, 4 marked scoreable-but-no-beliefs, **zero beliefs on any world-forward event**. All 108 beliefs in the DB are on `aeolus:` events (the weather discovery path, which bypasses the registry check because `aeolus_bucket_edges` constructs its own canonical event IDs and beliefs internally, `daemon.rs:2367–2426`).

The 4 "scoreable" watch events (unscoreable=false) are a historical artifact from a prior registry state or a prior code path; none has a belief regardless. They will never resolve without beliefs.

### Market-back discovery (confirmed stub / inert for match-before-create)

The market-back step runs with the live Kalshi catalog (T4.2 is wired, `daemon.rs:1924`). The prefilter applies `category_allowlist = ["weather"]` (`daemon.rs:1982`). Because the live catalog's weather markets already have edges (they were wired by Aeolus F7), the `current_edges_for_market` dedup check skips them (`daemon.rs:1993`), so no survivor reaches `market_back_discovery`.

Critically, the `existing_events` slice passed to `market_back_discovery` is always `Vec::new()` (`daemon.rs:2010–2011`). This means the "match-before-create" logic inside `market_back_discovery` (`discovery.rs:392–415`) can never fire — every normalization entry that names a `matches_event_id` is dropped as a hallucinated match (`discovery.rs:409–414`), and every survivor that should create a new event goes through the new-event draft path instead. No `01EVT` events exist in `fortuna_demo`, confirming this path is currently inert.

### Dedup correctness (confirmed adequate for current scale)

The `find_world_forward_duplicate_event` function (`daemon.rs:1651–1688`) applies Jaccard similarity ≥0.55 (or ≥0.35 with same-category AND same-benchmark), exact normalized-text match, and family-key match against the last 200 events by `created_at DESC`. At 20 current watch events this is fully covered. The family-key tests (`daemon.rs:4716–4725`) pass. The dedup correctly catches rephrased FOMC/Miran/wind variants in the current batch (evidenced by the `dead` status on duplicates in the DB). The `LIMIT 200` boundary becomes a risk only after the event table grows large.

### Operator seeding (confirmed: no path exists)

`fortuna-cli/src/main.rs` provides 8 commands (status/halt/rearm/kill/config check/logs/start/stop). None creates, lists, or modifies events. The only way for an operator to seed an event today is:
1. Raw `psql`: `INSERT INTO events (event_id, statement, resolution_criteria, resolution_source, benchmark_at, category, created_at) VALUES (...)`
2. Register a new source and hope world-forward synthesizes the event
3. Wire Aeolus weather (works, but only for weather markets)

The `EventsRepo::create` call at `repos.rs:443` is the right primitive; it just has no CLI surface.

### Category chaos (confirmed)

Twelve distinct category strings across 20 watch events. The `watchlist_schema()` at `discovery.rs:534` specifies `"category": {"type": "string"}` with no `"enum"` constraint. The model freely invents case variants and multi-level hierarchies (`macro/regulatory`, `macro/monetary-policy`, `Macro/Fed`). The `prefilter.category_allowlist` does exact-string comparison (`discovery.rs:129`), so a market filed under `Macro/Fed` will not pass a filter for `macro`.

## Self-Adversarial Pass

**Finding 1 (unscoreable resolution sources) severity P1**: Is this really P1 if the paper demo only needs Aeolus weather beliefs to work? The synthesis arm trades on `aeolus:` events with confirmed edges and open beliefs, and 108 such beliefs exist. So the paper loop IS closing on weather events. The P1 severity stands because world-forward beliefs are the stated mechanism for macro/news calibration (spec 5.12), the loop is running and spending budget, and the operator sees candidates that look active but produce nothing. A demo where every world-forward event shows 0 beliefs is confusing at best, demo-breaking at worst.

**Finding 2 (empty existing_events) severity P2**: Is this actually a BLOAT-cut rather than BLOCKS? For the current demo, which is weather-only, match-before-create would not help — there are no existing macro events to match against. Downgraded from P1 to P2 for this reason. But it is a genuine `BLOCKS` for any non-weather discovery.

**Finding 3 (dedup LIMIT 200)**: With only 20 watch events in the DB and a `dead` marker removing resolved events, this is not a live issue today. P3 is the right severity. The concern is forward-looking.

**Finding 4 (no operator seeding CLI)**: This is a genuine gap. P2 is appropriate: the demo works without it (events come from F7 weather), but the operator cannot prime specific macro events, cannot verify what events exist, and cannot debug the pipeline.

**What I may have missed**: I did not trace the full belief→edge→synthesis arm→proposal path for Aeolus weather events to verify it closes end-to-end. That path is covered by Area 8 (cognition). I also did not audit `aeolus_bucket_edges` internals (the F7 seam) in detail — that would be in scope for an F7-specific area.

## Open Questions for the Lead

1. **Resolution source matching**: Should `resolution_source` in the world-forward schema be an enum of registry source IDs (tight coupling but machine-verifiable), or should the registry adopt a domain-tag approach where a declared source like "Federal Reserve Board press releases" is classified under the `macro` domain tag and scoreable if ANY enabled source with tag `macro` exists? The current exact-match design is provably broken with real Opus output.

2. **Operator event seeding priority**: For the demo, is it acceptable to ask operators to use raw psql to seed macro events? Or should `fortuna event seed` be built before the demo readiness gate?

3. **Empty existing_events**: The comment at `daemon.rs:2007` says "the richer match-before-create events query is a follow-up (ledgered)". Is there a GAPS.md entry for this? If market-back is supposed to become the primary discovery path (it runs every segment before synthesis refresh), this stub significantly limits its utility.

4. **Category enum**: Should category be enforced by a DB CHECK or at the application layer? If it's application-layer (via schema enum), what is the canonical set? The current mix of `macro`, `Macro/Fed`, `monetary_policy`, `economics`, `economy` suggests the canonical set was never agreed.

5. **World-forward as demo signal**: The demo ROTA shows "0 beliefs on watch events". Should this be treated as a known limitation to communicate to viewers, or fixed before the demo?
